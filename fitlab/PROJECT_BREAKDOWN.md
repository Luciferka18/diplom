# FitLab — разбор проекта по полочкам

## 1) Что это за проект
- Это backend на **Laravel**, который отдает JSON API для фитнес-платформы: главная, программы, тренеры, магазин, корзина, заказы, авторизация. 
- В `README.md` отдельно описана целевая связка с frontend на Next.js и вариант auth через Sanctum bearer token.

## 2) Как устроен backend (верхний уровень)

### Точки входа
- Основной API-роутинг находится в `routes/api.php`.
- Все эндпоинты сгруппированы вокруг контроллеров из `app/Http/Controllers/Api/*`.

### Основные доменные сущности
- Пользователи: `User`
- Каталог тренировок: `Program`
- Тренеры: `Trainer`
- Товары: `Product`
- Заказы и корзина: `Order`, `OrderItem`
- Отзывы: `Review`
- Локации: `GymLocation`
- Обратная связь: `ContactRequest`

## 3) Реально подключенные API-эндпоинты

### Публичные
- `GET /api/home`
- `GET /api/trainers`, `GET /api/trainers/{id}`
- `GET /api/programs`, `GET /api/programs/{id}`
- `GET /api/products`, `GET /api/products/{id}`
- `POST /api/orders`
- `POST /api/auth/register`
- `POST /api/auth/login`

### Защищенные (`auth:sanctum`)
- `POST /api/auth/logout`
- `GET /api/cart`
- `POST /api/cart/items`
- `PATCH /api/cart/items/{itemId}`
- `DELETE /api/cart/items/{itemId}`
- `POST /api/cart/checkout`

## 4) Ключевые сценарии

### Авторизация
- Регистрация/логин выдают токен.
- Защищенные ручки работают по `Authorization: Bearer <token>`.

### Витрина
- `/api/home` возвращает агрегат: базовая информация о зале + подборки тренеров/программ/товаров/отзывов/локаций.

### Заказы
- Гостевой заказ через `POST /api/orders`.
- Авторизованная корзина хранится как `Order` со статусом `cart`.
- На checkout корзина переводится в обычный заказ со статусом `new`.

## 5) Что уже хорошо
- Базовая предметная модель закрывает core-сценарии (каталог + покупка + auth).
- В API есть разделение публичных и приватных методов.
- Для ключевых API есть feature-тесты (`tests/Feature/ApiEndpointsTest.php`).
- Есть сидеры для быстрого наполнения витрины (`database/seeders/*`).

## 6) Что стоит улучшить в первую очередь
1. **Согласовать статусную модель заказа**
   - Сейчас корзина и заказ используют одну таблицу `orders` через поле `status`.
   - Рекомендуется зафиксировать конечный автомат статусов (например: `cart -> new -> paid -> shipped -> done` и отдельно `cancelled`).

2. **Добавить API для уже созданных контроллеров**

3. **Усилить валидацию и бизнес-ограничения**
   - Проверка остатков/доступности товара перед checkout.
   - Защита от повторного checkout одного и того же cart-order.

4. **Расширить тестовое покрытие**
   - Негативные сценарии корзины/checkout.
   - Тесты на гостевой order с невалидными товарами/количеством.
   - Тесты авторизации на logout и неавторизованный доступ к cart.

5. **Подготовить API-контракт для frontend**
